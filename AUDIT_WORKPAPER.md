## Bugs Found

### FINDING-1: S-DEAD-1 — `FileWriteArgs` dead export
- **Code**: S-DEAD
- **Assertion**: `FileWriteArgs` struct is `pub` but has zero external consumers
- **Procedure**: Cross-referenced `pub` declaration against all imports across codebase
- **Evidence**: `capabilities/mod.rs:18` re-exports `FileWrite` but NOT `FileWriteArgs`. Zero files import `FileWriteArgs`
- **Conclusion**: FAIL
- **Trigger**: External consumer tries `use runtimo_core::capabilities::FileWriteArgs` — name not available despite being `pub`
- **Impact**: Misleading public API surface; dead type in docs
- **Recommendation**: Change `pub struct FileWriteArgs` to `struct FileWriteArgs` (private) or `pub(crate) struct FileWriteArgs`
- **Location**: core/src/capabilities/file_write.rs:67

### FINDING-2: S-DEAD-2 — `top_by_cpu()` and `top_by_mem()` dead exports
- **Code**: S-DEAD
- **Assertion**: `ProcessSnapshot::top_by_cpu()` and `top_by_mem()` are `pub` but only called internally
- **Procedure**: Grepped for `top_by_cpu` and `top_by_mem` across codebase
- **Evidence**: Only callers are `ProcessSnapshot::print_report()` at processes.rs:202 and :218 (same module). Zero external callers
- **Conclusion**: FAIL
- **Trigger**: External consumer tries to call these methods — they exist but serve no external purpose
- **Impact**: Unnecessary public API surface
- **Recommendation**: Change to `pub(crate)` or private methods
- **Location**: core/src/processes.rs:144,155

### FINDING-3: S-ORPHAN-1 — "ps aux" documentation stale
- **Code**: S-ORPHAN
- **Assertion**: Module doc claims "ps aux" but code uses `ps -eo pid,ppid,user,%cpu,%mem,vsz,rss,stat,start,time,comm`
- **Procedure**: Compared docstring claims against actual command at processes.rs:113
- **Evidence**: Doc at lines 4, 43, 88 says "ps aux". Code at line 113 uses `ps -eo` with explicit format (includes PPID, different column order)
- **Conclusion**: FAIL
- **Trigger**: Developer reads docs expecting BSD-style `ps aux` format, finds POSIX-style `ps -eo` instead
- **Impact**: Misleading documentation; wrong mental model for contributors
- **Recommendation**: Change "ps aux" to "ps with explicit format" in all three locations
- **Location**: core/src/processes.rs:4,43,88

### FINDING-4: S-ORPHAN-2 — "No hardcoded service names" contradicted
- **Code**: S-ORPHAN
- **Assertion**: Module doc says "No hardcoded service names" but `detect_service_for_port()` hardcodes port→name mapping
- **Procedure**: Compared docstring at telemetry.rs:8 against detect_service_for_port() at telemetry.rs:502-541
- **Evidence**: `match port { 22 => "ssh", 80|443 => "nginx", 3306 => "mysql", ... }` — all names hardcoded
- **Conclusion**: FAIL
- **Trigger**: Developer reads "No hardcoded service names" and expects fully dynamic detection
- **Impact**: Misleading design intent documentation
- **Recommendation**: Change to "No assumed running services" or "Service identification via well-known port mapping"
- **Location**: core/src/telemetry.rs:8

### FINDING-5: S-ORPHAN-3 — "Scans for known service processes" stale
- **Code**: S-ORPHAN
- **Assertion**: ServiceInfo doc says "Scans for known service processes" but code scans TCP ports
- **Procedure**: Compared doc at telemetry.rs:126 against implementation
- **Evidence**: Implementation uses `ss -ltnp` (TCP port scan), not `pgrep` (process scan). Zero process scanning in ServiceInfo::capture()
- **Conclusion**: FAIL
- **Trigger**: Developer expects process-based detection, finds port-based detection
- **Impact**: Wrong implementation model in docs
- **Recommendation**: Change to "Scans for listening TCP ports and maps to known services"
- **Location**: core/src/telemetry.rs:126

### FINDING-6: S-PROMISE-1 — vllm backwards compat fields never consumed
- **Code**: S-PROMISE
- **Assertion**: `vllm_version`, `vllm_running`, `vllm_port_bound` are computed but never read by any Rust code
- **Procedure**: Grepped for `services.vllm_version`, `services.vllm_running`, `services.vllm_port_bound` across all `*.rs` files
- **Evidence**: Zero matches in production code. Only present in struct definition and assignment. JSON serialization only
- **Conclusion**: FAIL
- **Trigger**: Toggling these fields would change JSON output but no Rust code behavior
- **Impact**: Dead computation; backwards-compat burden with no internal consumer
- **Recommendation**: Document as "JSON backwards-compat only" or remove if no external consumers exist
- **Location**: core/src/telemetry.rs:136-140
